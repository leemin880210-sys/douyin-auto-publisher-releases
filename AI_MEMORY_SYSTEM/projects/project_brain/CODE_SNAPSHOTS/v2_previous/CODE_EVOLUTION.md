# Code Evolution Memory（代码演进日志）

## 作用
记录本项目所有关键代码变更，保证 Codex 在每次修改时可以追踪历史演进，避免重复修改或结构断裂。

---

## 规则

1. 每次修改代码必须追加一条记录。
2. 不允许覆盖历史记录。
3. 必须包含：
   - 修改时间
   - 修改内容
   - 修改原因
   - 修改文件路径
   - 修改前状态
   - 修改后状态
4. 每一个代码修改记录必须维护“最近3次完整代码版本快照”。
5. 代码快照必须使用统一的 `Code Snapshot History` 格式。
6. 如果历史版本不足3次，缺失版本必须明确写为“无历史版本”。

---

## 📌 最近3次代码快照

系统必须为每一个代码修改记录保存：

### 1️⃣ 最新版本（v1 - 当前版本）

完整修改后的代码。

### 2️⃣ 上一版本（v2）

修改前最近一次代码。

### 3️⃣ 上上版本（v3）

再之前的一次代码。

---

## 🔁 Code Snapshot History

每条代码修改记录必须包含以下快照结构：

### v1（最新版本）

```text
完整代码片段
```

### v2（上一版本）

```text
完整代码片段；如果不存在，则写：无历史版本
```

### v3（上上版本）

```text
完整代码片段；如果不存在，则写：无历史版本
```

---

## 快照轮转规则

当发生新的代码修改时：

1. 新代码写入 v1。
2. 修改前的 v1 下移为 v2。
3. 修改前的 v2 下移为 v3。
4. 修改前的 v3 超出最近3次范围，不再保留在当前记录的快照区。
5. 历史记录本身不得覆盖或删除。

---

## 强制约束

- Codex 在修改任何代码前必须读取本文件
- 修改后必须追加记录
- 修改后必须维护最近3次完整代码版本快照
- 不允许跳过记录步骤
